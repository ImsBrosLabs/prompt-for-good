# Architecture Overview

## Components

### pfg-hub

The central server. Hosts the pre-qualified issue queue and coordinates runners.

**Responsibilities:**
- Periodic GitHub crawl to discover eligible open-source repositories
- Repository scoring (CI presence, test coverage, recent activity)
- Issue scoring (label quality, description clarity, estimated scope)
- FIFO dispatch queue: first available runner gets the next issue
- Runner registry with heartbeat tracking and token quota monitoring
- Contribution log (PR URLs, status, timestamps)

**Stack:** Kotlin · Spring Boot · PostgreSQL · GitHub REST API

**REST API:**
```
GET  /repos                    → list of qualified repositories
GET  /issues/next              → next available issue (FIFO)
POST /issues/{id}/claim        → runner claims an issue
POST /issues/{id}/done         → runner reports PR opened or failure
POST /runners/register         → register a new runner
POST /runners/{id}/heartbeat   → runner signals it is alive
GET  /stats                    → contribution dashboard data
```

**Database schema (simplified):**
```sql
repos         (id, github_url, language, score, last_crawled_at)
issues        (id, repo_id, github_id, title, score, status, claimed_by, claimed_at)
runners       (id, token, contributor_name, quota_remaining, last_seen_at)
contributions (id, issue_id, runner_id, pr_url, status, created_at)
```

**Issue status lifecycle:**
```
pending → claimed → done
                 → failed
```

### pfg-agent

The autonomous AI worker. Runs inside each pfg-runner container.

**Stack:** Python · LangChain · GitPython · Anthropic SDK · GitHub REST API

**7-Phase Pipeline:**

```
[1. Claim]   GET /issues/next from pfg-hub
             → receive issue metadata (title, body, repo URL, labels)

[2. Analyze] LLM: understand the problem
             → identify impacted files (max 5-10)
             → classify issue type (bug / feature / docs)

[3. Context] Clone repository locally (shallow clone)
             → extract only relevant files via grep/AST analysis
             → build a focused context window (< 100k tokens)

[4. Solve]   LLM: generate a targeted patch/diff
             → single-pass generation with structured output

[5. Verify]  Apply patch locally
             → run build + tests (language-specific runner)
             → on failure: retry with error feedback (max 3 attempts)
             → on persistent failure: abandon, report to hub

[6. PR]      git checkout -b pfg/{issue-id}
             → git commit + push
             → GitHub API: create PR
             → PR description generated by LLM (summary + approach + test results)

[7. Report]  POST /issues/{id}/done to pfg-hub
             → payload: pr_url, status (success|failed), tokens_used
```

**Default LLM:** `claude-3-5-sonnet` (Anthropic)

**Context strategy (anti-waste):**
- Issues pre-scored by pfg-hub → agent never analyzes un-solvable issues
- Shallow clone (depth=1) to avoid downloading full history
- File selection via AST analysis + grep before any LLM call
- Target context window: < 100k tokens per issue

### pfg-runner

The Docker container that contributors deploy.

**Stack:** Docker (Python base image, pfg-agent embedded)

**Configuration (`pfg.yaml`):**
```yaml
llm:
  provider: claude          # claude | openai | gemini (future)
  api_key: ${ANTHROPIC_API_KEY}
  model: claude-3-5-sonnet-20241022
  max_tokens_per_day: 100000

schedule:
  active_hours: "20:00-23:00"   # only run during these hours
  timezone: "Europe/Paris"

hub:
  url: https://hub.promptforgood.dev
  token: ${PFG_TOKEN}

contributor:
  name: "YourName"
  github_token: ${GITHUB_TOKEN}
```

## Data Flow

```
pfg-hub (cron job, every 6h)
  → GitHub API: search repos with good-first-issue, help-wanted labels
  → score each repo (has CI? has tests? recently active?)
  → score each open issue (description quality, scope estimate, labels)
  → insert qualified issues into DB with status=pending

pfg-runner (running on contributor machine)
  → poll pfg-hub GET /issues/next
  → pfg-agent executes 7-phase pipeline
  → pfg-hub POST /issues/{id}/done
```

## Key Design Decisions

| Decision | Rationale |
|---|---|
| Pre-qualification before dispatch | Avoid wasting LLM tokens on un-solvable or out-of-scope issues |
| FIFO dispatch | Simple, fair, no coordination overhead between runners |
| Build + tests must pass before PR | No noise PRs; only real contributions |
| Shallow clone | Minimize disk usage and clone time on contributor machines |
| Bring-your-own API key | Contributors control their spending; no central API key risk |
| Claude-first, plugin-friendly | Start with the best available model; allow OpenAI/Gemini later |
| Kotlin for pfg-hub | Said's primary language; strong typing, Spring Boot ecosystem |
| Python for pfg-agent | Best LangChain / LLM tooling ecosystem |

## Future Considerations

- **Embeddings-based context:** Use vector search over the codebase instead of grep for M2+
- **Language-specific build runners:** Detect language (Maven, Gradle, npm, pip, cargo) and adapt build commands
- **Issue difficulty tiers:** Easy / Medium / Hard scoring for better runner-issue matching
- **Multi-LLM support:** Plugin architecture to support OpenAI GPT-4o, Google Gemini
- **pfg-hub dashboard:** Web UI for stats (repos, PRs, contributors, token usage)
